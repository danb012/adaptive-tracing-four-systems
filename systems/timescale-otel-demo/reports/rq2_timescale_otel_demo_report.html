Research Question 2

RQ2 Runtime-Change Report: Timescale OpenTelemetry Demo

The Timescale OpenTelemetry Demo is a smaller password-generation microservice application with a strong observability pipeline. For RQ2 it provides a lighter-weight contrast to Train-Ticket.

RQ2: How do RL-based adaptive tracing methods behave under different runtime changes, such as latency spikes, throughput drops, and error bursts?

Q-Learning SARSA Bandit
Scenario count

4

Healthy baseline plus three explicit runtime-change scenarios.

Policies compared

3 RL methods

The RQ2 focus is on behavior inside the RL family rather than RL vs non-RL.

Reporting caveat

Important

Reward should be interpreted carefully here because the Timescale reward function is not a pure runtime-performance score.

Scenario Definitions

Healthy

Baseline application behavior with no added delay and no injected application error rate.

Latency Spike

Application delay is increased substantially while keeping the injected error rate at zero.

Error Burst

The injected application error rate is increased while delay is kept at the baseline.

Throughput Drop

Moderate added delay plus a mild error rate increase, intended to reduce achieved throughput.

Interpretation

This report focuses on how the three RL methods react when the runtime condition changes, not on whether RL beats non-RL baselines. The key outputs are the final latency, final throughput, observed error rate, selected tracing rate, and the number of clean decision windows.

The main things to look for are whether a policy raises tracing when the system degrades, whether it stabilizes after the change, and whether its runtime metrics remain controlled rather than oscillating or collapsing.

Reward should be interpreted carefully here because the Timescale reward function is not a pure runtime-performance score.
