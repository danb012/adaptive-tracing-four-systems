RQ1 Report for the Timescale OpenTelemetry Demo

This report evaluates RL-based and non-RL adaptive tracing policies on a lightweight password-generator microservice application. The goal is the same as in the Train-Ticket study: compare RL-based adaptive tracing with simpler non-RL baselines under healthy and faulted runtime conditions, using a common tracing backend and one shared experiment workflow.

1. Research Question

RQ1: How does RL-based adaptive tracing compare to non-RL adaptive tracing baselines, such as rule-based and clustering-based methods, when all methods are evaluated under the same microservice system and the same runtime conditions?

Literature gap addressed: direct side-by-side comparison between RL-based and non-RL adaptive tracing methods is still limited. This report extends the project beyond Train-Ticket and tests whether the comparison remains similar in a smaller, tracing-oriented microservice application.

2. The Timescale OpenTelemetry Demo

System description

The Timescale OpenTelemetry Demo is a lightweight password-generator microservice application designed as an observability playground. A generator service calls four downstream services to compose a password: upper, lower, digit, and special. A background load service continuously sends traffic to the generator.

This makes the system useful for adaptive tracing: it is small enough to run locally, but still produces real distributed traces across several services and request paths.

Tracing and storage path

The services send traces to the OpenTelemetry Collector. The collector forwards traces to Promscale, which stores them in TimescaleDB. Jaeger and Grafana sit on top of that tracing backend for exploration and inspection.

Services→ OTel Collector→ Promscale→ TimescaleDB→ Jaeger / Grafana

Application components

generatorcomposes the final password and fans out to the four downstream services
upperreturns random uppercase characters
lowerreturns random lowercase characters
digitreturns random digits
specialreturns random special characters

Compared with Train-Ticket, this system is smaller and easier to control. That makes it useful as a second testbed for the same research question.

3. Experimental Design

Policies evaluated

  • RL-based: q_learning, sarsa, bandit
  • Non-RL baselines: rule, kmeans

The action space was the same for every policy: tracing rates [0.05, 0.10, 0.20, 0.50, 0.80].

Scenarios

  • Healthy: APP_EXTRA_DELAY_MS=0, APP_ERROR_RATE=0
  • Faulted: APP_EXTRA_DELAY_MS=40, APP_ERROR_RATE=0.05

These controls were injected directly into the services so the same system could be run under stable and degraded runtime conditions.

Metrics

The report uses final cumulative run metrics collected from TimescaleDB root spans:

  • Average reward for RL policies only
  • Average sampling rate, plus minimum and maximum rate selected during the run
  • Final average latency in milliseconds
  • Final QPS
  • Final error rate
  • Total traces captured during the run

Each experiment ran for 20 seconds with a 5-second decision interval, producing four policy decisions per run.

6. Interpretation

Healthy condition

Under healthy conditions, the rule baseline produced the lowest final latency (133.85 ms) while keeping the sampling rate fixed at 0.05. Among the RL methods, q_learning was the strongest on runtime metrics because it combined low latency (193.02 ms), high QPS (2.392), zero final error rate, and a low fixed sampling rate.

sarsa was the only policy that materially changed its tracing rate in the healthy case. Its average rate rose to 0.1625 with a maximum of 0.50. That produced the highest RL average reward in healthy mode, but it also introduced a nonzero final error rate and a more aggressive sampling profile than the other methods.

Faulted condition

Under faulted conditions, the rule baseline again achieved the lowest final latency (211.29 ms) while staying at a low tracing rate. kmeans held the highest final QPS (2.275) at a fixed 0.10 rate, but with a higher error rate than the rule baseline.

Among the RL methods, bandit looked most stable on runtime behavior: it stayed at 0.05, kept moderate latency, and maintained better throughput than the other RL policies in the faulted case. q_learning achieved the highest RL average reward, but it also ended with the highest final latency in the whole faulted batch (949.49 ms), which weakens any claim that it was the best method overall on this system.

Reward-design caveat

The current reward function in this system adds positive terms for error_rate and avg_latency_ms instead of penalizing them. That means a higher RL reward does not necessarily imply better runtime performance here. This is important for reading the charts: q_learning appears strongest by reward in the faulted case, but that does not match the latency and throughput results.

Because of that, the strongest evidence for this system should come from the shared runtime metrics rather than RL reward alone.

7. Answer to RQ1 for the Timescale OpenTelemetry Demo

This experiment answers RQ1 for the Timescale OpenTelemetry Demo by providing a direct side-by-side comparison between RL-based and non-RL adaptive tracing methods in a second microservice system. The result is more mixed than the Train-Ticket result. On this system, the non-RL rule baseline produced the lowest final latency in both healthy and faulted runs, while bandit showed the most stable low-rate RL behavior. q_learning achieved the highest RL average reward in the faulted scenario, but that advantage is not supported by the final runtime metrics because the current reward formulation is not well aligned with performance quality.

The practical takeaway is that system behavior matters: the same research question can produce different winners on different microservice systems. This is exactly why adding more than one system strengthens the project. The Timescale OpenTelemetry Demo shows that RL-based adaptive tracing does not automatically dominate non-RL baselines once the application and observability stack change.

8. Limitations

Short runs

Each experiment had only four decisions. That is enough for an initial comparison, but not enough to claim strong RL convergence.

Reward asymmetry

Non-RL baselines do not emit reward, and the current RL reward function is not fully aligned with desired runtime quality. That makes reward-based comparison weaker than metric-based comparison.

Single-run variability

Healthy and faulted results do not always separate cleanly. That indicates this system still has enough runtime variance that repeated runs or longer windows would improve confidence.

Additional notes

  • Most policies remained at a fixed rate across the whole run, so the current comparison is closer to an initial adaptive-tracing benchmark than a mature long-horizon learning study.
  • The fault injection in the Timescale OpenTelemetry Demo is lighter than a full cascading failure scenario. It creates useful signal, but stronger fault cases would improve the study.
  • Trace usefulness was still not measured directly. The report evaluates runtime and tracing-control behavior, not whether the traces were better for debugging.

9. Conclusion

This report completed the RQ1 comparison for the Timescale OpenTelemetry Demo. The experiment evaluated three RL-based policies and two non-RL baselines under healthy and faulted conditions using the same tracing backend and the same policy workflow. The resulting evidence is useful for the overall project because it shows that the Train-Ticket conclusion does not transfer unchanged to another microservice application.

In this application, the rule baseline was strongest on final latency in both scenarios, while bandit looked like the most stable RL policy on runtime metrics. Q-Learning achieved the highest RL average reward in the faulted case, but that signal is weakened by the current reward design. Overall, this report still addresses the literature gap by adding another direct RL vs non-RL comparison, but it also highlights that adaptive tracing results are system-dependent and should not be generalized from one testbed alone.
