Research Question 2

RQ2 Runtime-Change Report: Grafana QuickPizza

Grafana QuickPizza is a pizza recommendation microservice demo with a Grafana-native local observability stack. For RQ2 it tests whether the RL policies still react coherently when the same adaptive tracing workflow is moved onto a fourth application.

RQ2: How do RL-based adaptive tracing methods behave under different runtime changes, such as latency spikes, throughput drops, and error bursts?

Scenario Definitions

Healthy

Baseline application behavior with standard POST traffic to /api/pizza and no injected delay or error headers.

Latency Spike

Request headers enforce deterministic recommendation delay to increase end-to-end latency without intentionally increasing the error rate.

Error Burst

Request headers inject deterministic ingredient-fetch failures that propagate back to the public API as observable error responses.

Throughput Drop

Worker count and pause interval are reduced to intentionally lower achieved throughput.

Interpretation

This report focuses on how the three RL methods react when the runtime condition changes, not on whether RL beats non-RL baselines. The key outputs are the final latency, final throughput, observed error rate, selected tracing rate, and the number of clean decision windows.

The main things to look for are whether a policy raises tracing when the system degrades, whether it stabilizes after the change, and whether its runtime metrics remain controlled rather than oscillating or collapsing.

QuickPizza should still be interpreted carefully because scrape health across the full telemetry stack is not yet completely clean, even though the corrected runtime-change matrix is now available.
