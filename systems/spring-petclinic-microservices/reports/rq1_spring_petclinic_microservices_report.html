1. Research Question

RQ1: How does RL-based adaptive tracing compare to non-RL adaptive tracing baselines, such as rule-based and clustering-based methods, when all methods are evaluated under the same microservice system and the same runtime conditions?

Why Petclinic matters: unlike the smaller password generator application, Spring Petclinic Microservices is a Spring Cloud business-style system with gateway routing, discovery, configuration, and independent domain services. It is a stronger realism check for the adaptive tracing workflow.

2. The Spring Petclinic Microservices Application

System description

Spring Petclinic Microservices is the distributed variant of the Spring Petclinic sample application. It exposes a public web UI through an API gateway and routes requests to separate business services for customers, visits, and vets. Supporting infrastructure includes a config server and a discovery server.

This is a useful third application for the study because it reflects a realistic service topology rather than a tracing-only demo. It also already includes observability primitives: Zipkin for tracing and Prometheus for application metrics.

Observability path

Tracing is exported to Zipkin and service metrics are scraped by Prometheus. For the Petclinic experiments, Prometheus metrics were used as the measurement source because they remained more stable than short-window trace counts at low sampling rates.

3. Experimental Design

Policies evaluated

  • RL-based: q_learning, sarsa, bandit
  • Non-RL baselines: rule, kmeans

The action set was the same as in the other applications: tracing probabilities [0.05, 0.10, 0.20, 0.50, 0.80].

Scenario definition

  • Healthy: normal generated gateway traffic
  • Faulted: first try Chaos Monkey latency on visits-service; if unsupported in the published image, fall back to a heavier degraded load profile

The Docker images used here did not reliably expose Chaos Monkey actuator endpoints, so the runner falls back to a stronger traffic profile instead of aborting the experiment.

Important measurement caveat

The current reward function is not a pure “better system performance” score. It still adds positive reward components for latency, error rate, and drift before subtracting trace-cost terms. That means average reward must be interpreted cautiously. Final latency, QPS, error rate, and trace totals are the stronger indicators for this application.

Important runtime caveat

The Petclinic runs are much shorter and noisier than the Train-Ticket study. Most policy runs produced only 1–2 decision windows, and two runs still contained one empty decision window. These results are usable as a first-pass Petclinic comparison, but they are weaker evidence than the longer, cleaner runs in the other two systems.

7. Interpretation

Main result

Bandit is the strongest overall Petclinic policy on final runtime metrics. It achieves the best final latency in both healthy and faulted conditions, the highest final QPS in both conditions, and the highest RL average reward in both conditions.

This is a different pattern from the Timescale OpenTelemetry Demo, where the rule baseline was strongest on final latency, and from Train-Ticket, where RL methods were comparatively more dominant.

Policy-specific observations

  • Bandit: best final latency and QPS in both conditions; one faulted decision window is empty, but the final status is still the strongest.
  • Q-Learning: valid nonzero runs, but weaker final QPS and higher latency than Bandit, especially in the healthy case.
  • SARSA: conservative low-rate behavior; stronger than Q-Learning on healthy throughput, but weaker than Bandit in both scenarios.
  • Rule: expensive in the faulted run (avg rate 0.65) without producing the best runtime outcome.
  • K-Means: clean and stable final values, but generally lower throughput than Bandit and weaker healthy latency.

Why the faulted baseline looks “better” than the healthy baseline

The fixed-rate faulted baseline has higher throughput and lower final latency than the fixed-rate healthy baseline. That is not evidence that the application becomes healthier when degraded. It is a result of the fallback scenario design used in this stack: when Chaos Monkey is unavailable, the faulted run uses a heavier traffic profile. For that reason, the Petclinic interpretation is primarily about relative policy behavior inside the same scenario, not absolute healthy-vs-faulted superiority.

Strength of evidence

The Petclinic result set is complete, but it is less mature than the Train-Ticket and Timescale result sets. The policy runs are short, the decision counts are low, and a small number of windows are empty. The report should therefore present Petclinic as a supporting third application, not as the strongest empirical anchor of the study.

8. Answer to RQ1 for Petclinic

On Spring Petclinic Microservices, the comparison does not reproduce the same ranking observed in the other applications. Bandit is the strongest policy on final runtime metrics, achieving the best final latency and best final throughput in both healthy and degraded runs. Q-Learning and SARSA remain viable RL policies, but neither dominates Bandit on the final observed metrics. The non-RL baselines are mixed: K-Means is stable but slower, while Rule becomes expensive in the degraded run without converting that higher tracing rate into better outcomes.

The Petclinic result therefore strengthens the overall dissertation argument: the effectiveness of adaptive tracing is system-dependent. The same policy does not win on every application, and a direct RL-vs-non-RL comparison is necessary rather than assumed.

9. Limitations

  • The reward function is not yet aligned with a pure performance objective; reward values should be treated as secondary evidence.
  • The faulted scenario falls back to heavier traffic when Chaos Monkey endpoints are unavailable in the published Docker images.
  • Most policy runs have only one or two decision windows, so adaptation depth is limited.
  • Some runs contain one empty decision window, which slightly weakens the stability of the result set.
  • For a stronger final study, Petclinic would benefit from longer runs and repeated trials per policy.
