Research Question 2

RQ2 Runtime-Change Report: Spring Petclinic Microservices

Spring Petclinic Microservices is the business-domain application in the study. For RQ2 it shows whether RL policies react consistently when the runtime signal is noisier and decision windows are longer.

RQ2: How do RL-based adaptive tracing methods behave under different runtime changes, such as latency spikes, throughput drops, and error bursts?

Q-Learning SARSA Bandit
Scenario count

4

Healthy baseline plus three explicit runtime-change scenarios.

Policies compared

3 RL methods

The RQ2 focus is on behavior inside the RL family rather than RL vs non-RL.

Reporting caveat

Important

This system is the least stable of the three. Low decision counts or sparse windows should be treated as a reporting limitation, not ignored.

Scenario Definitions

Healthy

Baseline operating condition with the standard gateway traffic profile.

Latency Spike

Best-effort latency injection on the visits service, with a traffic fallback if Chaos Monkey is unavailable.

Error Burst

Best-effort exception injection on the visits service, with the existing fallback behavior if Chaos Monkey endpoints are unavailable.

Throughput Drop

A deliberately slower traffic profile intended to reduce throughput even if direct chaos hooks are unavailable.

Interpretation

This report focuses on how the three RL methods react when the runtime condition changes, not on whether RL beats non-RL baselines. The key outputs are the final latency, final throughput, observed error rate, selected tracing rate, and the number of clean decision windows.

The main things to look for are whether a policy raises tracing when the system degrades, whether it stabilizes after the change, and whether its runtime metrics remain controlled rather than oscillating or collapsing.

This system is the least stable of the three. Low decision counts or sparse windows should be treated as a reporting limitation, not ignored.
